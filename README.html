Flask Mailer API

A minimal Flask endpoint for sending contact-form email through Gmail SMTP with strict validation, browser-origin checks, in-memory throttling, plain-text message construction, and support for multiple configured recipients.

Important: this version does not use Redis or reCAPTCHA. That keeps setup simple, but it also means abuse resistance is best-effort only. A determined attacker can still script requests from outside the browser. If you need strong public-internet protection, add a server-side trust signal such as CAPTCHA, auth, or a backend relay.

1. What It Does

  • Exposes one endpoint: POST /send-mail.
  • Accepts JSON fields: name, email, phone, subject, and message.
  • Sends mail using Gmail SMTP and an App Password.
  • Delivers each submission to one or more configured recipient inboxes.
  • Uses the SMTP account as From and the visitor email as Reply-To.
  • Uses Bcc with an undisclosed-recipients header so internal addresses are not exposed to one another.
  • Rejects malformed requests, oversized bodies, bad origins, empty user agents, and rapid replay attempts.

2. Requirements

  • Python 3.11 or newer recommended.
  • A Gmail account with 2-Step Verification enabled.
  • A Gmail App Password for SMTP.
  • A single-process deployment. Do not run multiple workers without adding shared state.

3. Setup Guide

Step 1: Create a virtual environment

python -m venv .venv

Step 2: Activate it

Windows PowerShell:
.venv\Scripts\Activate.ps1

macOS / Linux:
source .venv/bin/activate

Step 3: Install dependencies

pip install -r requirements.txt

Step 4: Configure environment variables

copy .env.example .env

Then edit .env with your real values.

Step 5: Run the API locally

flask --app app run

The endpoint will be available at http://127.0.0.1:5000/send-mail.

4. Environment Variables

VariableRequiredDescription
DEFAULT_SUBJECTNoFallback subject when the client does not send one.
ALLOWED_ORIGINSYesA comma-separated list of exact frontend origins allowed by CORS, for example https://www.example.com,https://portfolio.example.com.
SMTP_EMAILYesThe Gmail address used to log in and send mail.
SMTP_PASSWORDYesYour Gmail App Password. Never commit this value.
RECEIVER_EMAILSYesA comma-separated list of recipient inboxes, for example inbox@example.com,team@example.com. Up to 20 validated addresses are accepted.
TRUST_PROXY_COUNTNoNumber of trusted reverse proxies in front of Flask. Set this correctly if you use Nginx, a load balancer, or a platform proxy.
LOG_LEVELNoLogging level such as INFO or WARNING.

5. Request Format

{
  "name": "Jane Doe",
  "email": "jane@example.com",
  "phone": "+1 555 555 5555",
  "subject": "Project inquiry",
  "message": "Hello, I would like to discuss a build."
}

Rules:

  • name is required and capped at 100 characters.
  • email is required and normalized to lowercase.
  • phone is required, capped at 20 characters, and limited to a small safe character set.
  • subject is optional and defaults to DEFAULT_SUBJECT.
  • message is required and capped at 1000 characters.
  • Recipients are not accepted from the client. They come only from the server environment.

6. Frontend Example

async function sendContactForm(payload) {
  const response = await fetch("https://api.example.com/send-mail", {
    method: "POST",
    headers: {
      "Content-Type": "application/json",
      "Accept": "application/json"
    },
    body: JSON.stringify(payload),
    credentials: "omit"
  });

  const data = await response.json().catch(() => ({}));
  if (!response.ok) {
    throw new Error(data.error || "request_failed");
  }
  return data;
}

Do not set a restrictive Referrer-Policy that strips the Referer header, because the API uses it as part of its browser-origin checks in production mode.

7. Security Controls

  • Strict CORS response for one configured origin only.
  • Request rejection when Origin is wrong.
  • Production rejection when Referer does not match the specific allowed origin that sent the request.
  • Rejection when User-Agent is missing.
  • Rejection when the request is not application/json.
  • Strict schema validation with unexpected keys rejected.
  • CRLF and control-character blocking for header-safe fields.
  • Plain-text email body only.
  • Recipient list validation and deduplication at startup.
  • Bcc delivery so recipient addresses are not leaked in message headers.
  • Per-IP rate limiting through flask-limiter.
  • In-memory cooldown after repeated invalid submissions.
  • Replay suppression for duplicate payloads over a short window.
  • No sensitive request content in logs.

8. Production Deployment

Because Redis was removed, run one process only. A safe example is a single Gunicorn worker with threads:

gunicorn --workers 1 --threads 8 --bind 0.0.0.0:8000 app:app

Recommended production shape:

  • Put Nginx or another reverse proxy in front.
  • Terminate HTTPS at the proxy.
  • Forward the real client IP and set TRUST_PROXY_COUNT correctly.
  • Do not expose Flask's development server directly to the internet.
  • Store secrets in environment variables or your platform secret manager.
  • Keep the recipient list in RECEIVER_EMAILS server-side only.

10. Known Limitations

  • No reCAPTCHA means there is no strong human-verification step.
  • No Redis means rate limits and replay protection are only per process.
  • Origin and referer checks help in browsers, but custom clients can spoof them.
  • Without authentication or a backend proxy, determined abuse cannot be fully prevented.

11. Hardening Checklist

  • Use HTTPS in production.
  • Set exact ALLOWED_ORIGINS values.
  • Use a Gmail App Password, never your normal account password.
  • Keep RECEIVER_EMAILS limited to trusted internal inboxes only.
  • Run a single worker unless you add shared state.
  • Review logs for repeated mailer_rejected events.
  • Rotate SMTP credentials if you suspect abuse.
  • Add CAPTCHA or a backend broker later if abuse becomes a problem.
